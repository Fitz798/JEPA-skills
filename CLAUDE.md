# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

JEPA Skills is a reusable skill package for Claude Code agents doing JEPA (Joint Embedding Predictive Architecture) research. It provides a structured skill definition (`jepa-research/SKILL.md`), metadata indexes for papers and code, validation/build scripts, and a full evaluation benchmark suite. The current version is `v1.2.1`.

## Common commands

All commands run from repo root. Only Python stdlib is required (3.11+).

### Metadata build and validation

```powershell
# Rebuild paper catalog from source index
python scripts/build_paper_catalog.py --input metadata/papers_index.csv --output metadata/paper_catalog.csv

# Validate evidence index (add --check-paths when corpus/ assets are on disk)
python scripts/validate_evidence_index.py --input metadata/evidence_index.csv --check-paths --json eval/results/evidence_index_validation.json

# Check metadata completeness for release gating
python scripts/check_metadata_completeness.py --input metadata/paper_catalog.csv --json eval/results/metadata_completeness_report.json
```

### Citation candidate generation

```powershell
python jepa-research/scripts/make_citation_candidates.py --metadata metadata/paper_catalog.csv --query "multimodal,vision-language,audio,transfer" --topk 20 --output eval/results/rq05_candidates.csv
```

### Eval workflow

```powershell
# Extract one mode's prompts from the 40-prompt set
python eval/filter_eval_by_mode.py --input eval/test_prompts_v1.jsonl --mode research_qa --output eval/results/research_qa_prompts_v1.jsonl

# Summarize judged results
python eval/summarize_eval.py --input eval/results/<scored>.jsonl --output eval/results/<summary>.md

# Check evidence/mapping coverage in response markdown
python eval/check_evidence_coverage.py --responses eval/results/<responses>.md --output eval/results/<coverage>.json
```

### Sanity check after moving repo

```powershell
python scripts/build_paper_catalog.py --input metadata/papers_index.csv --output metadata/paper_catalog.csv
python scripts/validate_evidence_index.py --input metadata/evidence_index.csv --check-paths --json eval/results/evidence_index_validation.json
python scripts/check_metadata_completeness.py --input metadata/paper_catalog.csv --json eval/results/metadata_completeness_report.json
```

## Architecture

### Three-layer design

1. **Skill layer** (`jepa-research/`) — the agent-facing contract
   - `SKILL.md`: 4 modes (research_qa, reproduction_plan, debug_triage, writing_citation) with output templates and evidence-binding requirements
   - `references/evidence_binding_rules.md`: minimum evidence counts per mode, mapping format, missing-evidence handling
   - `references/output_templates.md` and `task_taxonomy.md`: template and task classification reference
   - `scripts/make_citation_candidates.py`: ranked paper retrieval using metadata signals, query-term expansion, and JEPA-family preference scoring

2. **Metadata layer** (`metadata/`) — structured data backing every answer
   - `papers_index.csv` (~148 papers): source of truth for paper inventory (raw index with filesystem paths, sha256, tier labels)
   - `paper_catalog.csv`: built artifact from `papers_index.csv` with added taxonomy fields (modality, task, evidence_level, code_id, code_link_status, link_confidence). Generated by `scripts/build_paper_catalog.py`
   - `evidence_index.csv` (19 rows): production evidence records keyed by `evidence_id` (E0001-E0018) with type, locator, claim_summary, confidence, and status. Schema in `docs/evidence-index-schema.md`
   - `code_index.csv`: local code archive inventory

3. **Eval layer** (`eval/`) — regression gate for skill changes
   - `test_prompts_v1.jsonl`: 40 prompts across 4 modes (10 each)
   - `scoring_rubric.md`: 5-dimension rubric (factual_correct, citation_hit, executable, debug_hit, hallucination)
   - `results/`: judged JSONL files + markdown summaries + benchmark comparisons
   - Key scripts: `summarize_eval.py`, `filter_eval_by_mode.py`, `check_evidence_coverage.py`

### Data flow

```
papers_index.csv  ──build_paper_catalog.py──▶  paper_catalog.csv
                                                        │
                              make_citation_candidates.py
                                                        │
evidence_index.csv  ──validate_evidence_index.py──▶  validation JSON
                              │
                              ▼
                         SKILL.md  (agent reads metadata CSVs + evidence index at answer time)
```

### Key conventions

- **Evidence IDs**: `E0001`-`E0018` in `evidence_index.csv` are stable. Do not renumber existing rows. New evidence gets the next available ID.
- **Paper tiers**: `core` (~11 papers) vs `long_tail` (~137 papers). Core papers define the evaluation scope; long-tail papers are supplementary. Release gates only check core metadata completeness.
- **Paper-code linking policy** (`docs/paper-code-linking-policy.md`): conservative — `exact_local_match` only when the archive is confirmed as the paper's official implementation. Otherwise use `no_known_local_code`, `family_match_only`, or `multiple_local_candidates`.
- **Confidence markers**: `high` (title-page or abstract-level verification), `medium` (likely but not directly confirmed), `low` (inferred). Mark uncertain evidence rows as `provisional` or `missing_locator` status.
- **Missing evidence**: When a claim has no supporting evidence, use `missing_evidence` in the mapping rather than inventing references. This is enforced by `check_evidence_coverage.py`.

### Response contract (all modes)

Every non-trivial answer must include:
1. **Action/Claim list** with IDs (C1, C2, ... or A1, A2, ...)
2. **Evidence table** with IDs (E1, E2, ...) each with source type and pointer
3. **Mapping line**: `C# -> E#` or `A# -> E#`
4. **Confidence**: `high` | `medium` | `low`

Minimum evidence per mode: research_qa ≥ 2 papers; reproduction_plan ≥ 1 paper + 1 code/protocol; debug_triage ≥ 1 code/protocol; writing_citation every claim → at least one paper.

## Release process

A release is complete when:
1. `metadata/paper_catalog.csv` rebuilt cleanly
2. `metadata/evidence_index.csv` passes validation
3. `metadata_completeness_report.json` has `core_gate_ok: true`
4. All judged eval slices have current summaries
5. Benchmark comparison doc exists with measured deltas
6. CHANGELOG.md and VERSION updated
7. Git tag created matching VERSION

Full workflow documented in `docs/v1.2.0-release-eval-workflow.md`.
