# v1.2.0 Release And Eval Workflow

This document defines the minimum rerunnable workflow for a public `v1.2.0` release.

## Inputs

- metadata source: `metadata/papers_index.csv`
- evidence source: `metadata/evidence_index.csv`
- eval prompts: `eval/test_prompts_v1.jsonl`
- judged baselines:
  - `eval/results/baseline_v1_0_0_full40_scored.jsonl`
  - `eval/results/v1_1_0_full40_scored.jsonl`

## Step 1: Rebuild metadata artifacts

Rebuild the paper catalog:

```powershell
python scripts/build_paper_catalog.py --input metadata/papers_index.csv --output metadata/paper_catalog.csv
```

Validate evidence:

```powershell
python scripts/validate_evidence_index.py --input metadata/evidence_index.csv --check-paths --json eval/results/evidence_index_validation.json
```

Check metadata completeness:

```powershell
python scripts/check_metadata_completeness.py --input metadata/paper_catalog.csv --json eval/results/metadata_completeness_report.json
```

Expected outputs:

- `metadata/paper_catalog.csv`
- `eval/results/evidence_index_validation.json`
- `eval/results/metadata_completeness_report.json`

Release gate for this step:

- evidence validation passes
- core metadata completeness gate passes

## Step 2: Build QA-only eval slice

Extract the `research_qa` prompts:

```powershell
python eval/filter_eval_by_mode.py --input eval/test_prompts_v1.jsonl --mode research_qa --output eval/results/research_qa_prompts_v1.jsonl
```

Expected output:

- `eval/results/research_qa_prompts_v1.jsonl`

## Step 3: Summarize judged runs

Baseline QA summary:

```powershell
python eval/filter_eval_by_mode.py --input eval/results/baseline_v1_0_0_full40_scored.jsonl --mode research_qa --output eval/results/baseline_v1_0_0_research_qa_scored.jsonl
python eval/summarize_eval.py --input eval/results/baseline_v1_0_0_research_qa_scored.jsonl --output eval/results/baseline_v1_0_0_research_qa_summary.md
```

Current QA summary:

```powershell
python eval/filter_eval_by_mode.py --input eval/results/v1_1_0_full40_scored.jsonl --mode research_qa --output eval/results/v1_1_0_research_qa_scored.jsonl
python eval/summarize_eval.py --input eval/results/v1_1_0_research_qa_scored.jsonl --output eval/results/v1_1_0_research_qa_summary.md
```

Expected outputs:

- `eval/results/baseline_v1_0_0_research_qa_scored.jsonl`
- `eval/results/baseline_v1_0_0_research_qa_summary.md`
- `eval/results/v1_1_0_research_qa_scored.jsonl`
- `eval/results/v1_1_0_research_qa_summary.md`

## Step 4: Write comparison artifact

Produce a release-facing comparison note that combines:

- QA metrics
- evidence validation status
- metadata completeness status
- any regressions or non-improvements

Current expected output filename:

- `eval/results/v1_2_0_benchmark_comparison.md`

## Release Checklist

- `metadata/paper_catalog.csv` rebuilt from source script
- `metadata/evidence_index.csv` validates cleanly
- `eval/results/metadata_completeness_report.json` exists and `core_gate_ok` is `true`
- `eval/results/research_qa_prompts_v1.jsonl` exists
- baseline and current QA summaries exist
- `eval/results/v1_2_0_benchmark_comparison.md` exists
- release notes explicitly state whether QA citation hit improved or stayed flat

## Current Readiness Notes

- `v1.2.0` is stronger on metadata structure and release hygiene than on judged QA gains.
- If QA citation hit does not improve on the historical judged slice, the release note must say that directly instead of implying a retrieval win.
