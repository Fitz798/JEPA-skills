# v1.2.0 Roadmap

This document defines the next delivery milestone for the `jepa-research` skill after `v1.1.0`.

## Release Position

`v1.1.0` established the response contract:

- evidence blocks are required
- action or claim to evidence mapping is required
- evaluation exists and shows strong citation-hit improvement

`v1.2.0` should convert that contract from mostly prompt-level discipline into a more data-backed and repeatable workflow.

## Release Goals

1. Make evidence binding depend less on manual answer formatting and more on structured metadata.
2. Improve `research_qa`, the weakest current mode by citation-hit rate.
3. Strengthen reproduction and debug outputs with clearer paper-to-code links.
4. Turn evaluation into a repeatable regression gate for future skill changes.

## Non-Goals

- Full PDF parsing pipeline for every local paper.
- Automated reproduction execution across third-party repositories.
- Web-backed retrieval or external service dependencies.
- Large prompt rewrite across all modes without evaluation coverage.

## Success Criteria

`v1.2.0` is complete when all of the following are true:

1. A real `metadata/evidence_index.csv` exists and is documented.
2. `paper_catalog.csv` can carry explicit paper-to-code linkage fields.
3. `research_qa` citation-hit improves from `80%` to at least `90%` on the current 10-prompt slice.
4. A reproducible evaluation command set is documented and produces machine-readable outputs.
5. At least one automated check fails when evidence mapping or required metadata fields are missing.

## Workstreams

## W1. Structured Evidence Data

Objective: replace the template-only evidence layer with a usable project asset.

Deliverables:

- `metadata/evidence_index.csv`
- documented schema and fill rules
- script support for validating required columns and row consistency

Why this is first:

- it unlocks stronger claim grounding
- it reduces prompt-only enforcement
- it creates a reusable substrate for every mode

## W2. Paper-Code Linking

Objective: make reproduction and debug advice point to concrete code artifacts, not only paper IDs.

Deliverables:

- expanded `paper_catalog.csv` fields for `code_id`, `code_repo_url`, `code_commit`, `modality`, `task`, `evidence_level`
- documented linking rules between papers and code archives
- consistency checks against `metadata/code_index.csv`

## W3. Research QA Retrieval Quality

Objective: improve how the skill chooses supporting papers for conceptual questions.

Deliverables:

- better metadata coverage for modality, task, and evidence level
- updated helper workflow for citation candidate generation
- evaluation improvements specifically on `research_qa`

## W4. Eval and Regression Gate

Objective: make future skill edits measurable and safer.

Deliverables:

- one documented evaluation workflow for baseline and comparison runs
- automated checks for evidence coverage and metadata completeness
- a release checklist for future tags

## Milestones

## M1. Data Backbone

Scope:

- finalize evidence schema
- create initial populated `evidence_index.csv`
- add validation script

Exit criteria:

- evidence file exists with real rows
- validation catches missing IDs, empty locators, and invalid evidence types

## M2. Linking and Retrieval

Scope:

- expand paper catalog fields
- map core papers to code artifacts
- improve citation candidate workflow

Exit criteria:

- at least all `core` papers with code have explicit links
- `research_qa` benchmark improves to target

## M3. Release Gate

Scope:

- document standard eval commands
- add checks for metadata completeness and evidence coverage
- produce `v1.2.0` comparison summary

Exit criteria:

- another contributor can run the checks from docs alone
- release notes can reference measured deltas

## Issue Breakdown

The issues below are ordered by dependency, not by desirability.

### JEPA-201: Define evidence index schema

Goal:
Create the production schema for `metadata/evidence_index.csv`.

Scope:

- decide required columns
- define allowed `evidence_type` values
- define locator format rules for paper, code, protocol, and log evidence
- document confidence semantics

Acceptance criteria:

- schema documented in repo
- template updated or replaced
- examples cover all supported evidence types

Dependencies:

- none

### JEPA-202: Create initial populated evidence index

Goal:
Replace the placeholder template with a real starter dataset.

Scope:

- add evidence rows for core JEPA papers
- add evidence rows for linked local code archives where available
- ensure each row has stable IDs and usable locators

Acceptance criteria:

- `metadata/evidence_index.csv` exists
- file has enough rows to support current eval prompts
- no duplicate evidence IDs

Dependencies:

- `JEPA-201`

### JEPA-203: Add evidence validation script

Goal:
Fail fast on malformed or incomplete evidence records.

Scope:

- validate headers
- validate non-empty required fields
- validate allowed `evidence_type`
- validate duplicate IDs and broken references

Acceptance criteria:

- script exits non-zero on invalid input
- script documented in `README` or release doc

Dependencies:

- `JEPA-201`

### JEPA-204: Expand paper catalog schema

Goal:
Make `paper_catalog.csv` useful for retrieval and reproduction workflows.

Scope:

- extend catalog columns
- update `scripts/build_paper_catalog.py`
- preserve compatibility with current index inputs

Acceptance criteria:

- regenerated catalog contains new linkage and taxonomy fields
- script remains runnable from repo root

Dependencies:

- none

### JEPA-205: Link papers to code artifacts

Goal:
Create explicit connections between paper records and local code assets.

Scope:

- add `code_id` or equivalent mapping field
- map core papers to archives in `metadata/code_index.csv`
- document how to mark uncertain or multiple candidates

Acceptance criteria:

- linked rows exist for all known core repos
- uncertain mappings are explicit, not implicit

Dependencies:

- `JEPA-204`

### JEPA-206: Improve citation candidate ranking

Goal:
Reduce weak supporting-paper selection in `research_qa`.

Scope:

- improve metadata features used by ranking
- update `jepa-research/scripts/make_citation_candidates.py` or add a companion script
- test conceptual query slices, not only keyword overlap

Acceptance criteria:

- ranking logic uses more than raw string containment
- sample outputs for representative research questions look defensible

Dependencies:

- `JEPA-204`
- optionally `JEPA-202`

### JEPA-207: Add research QA focused eval slice

Goal:
Make the weakest mode easier to iterate on.

Scope:

- isolate or document the 10-prompt `research_qa` slice
- define target metrics for citation-hit and factual correctness
- produce comparison output for this slice alone

Acceptance criteria:

- one command path exists for QA-only evaluation
- summary clearly compares before and after

Dependencies:

- none

### JEPA-208: Add metadata completeness checks

Goal:
Catch thin metadata before it degrades answer quality.

Scope:

- check coverage for modality, task, evidence level, and code linkage fields
- report missing values and coverage rates

Acceptance criteria:

- script outputs machine-readable report
- report is suitable for release gating

Dependencies:

- `JEPA-204`

### JEPA-209: Document the release/eval workflow

Goal:
Make `v1.2.0` repeatable by someone other than the original author.

Scope:

- baseline run instructions
- comparison run instructions
- required output artifacts
- release checklist

Acceptance criteria:

- doc can be followed from clean repo context
- expected outputs and filenames are specified

Dependencies:

- `JEPA-203`
- `JEPA-207`
- `JEPA-208`

### JEPA-210: Produce v1.2.0 benchmark comparison

Goal:
Close the release with measured deltas against `v1.1.0`.

Scope:

- run benchmark
- summarize metrics
- note wins, regressions, and unresolved gaps

Acceptance criteria:

- comparison artifact exists under `eval/results/`
- changelog-ready summary is available

Dependencies:

- `JEPA-202`
- `JEPA-205`
- `JEPA-206`
- `JEPA-209`

## Suggested Execution Order

1. `JEPA-201` schema
2. `JEPA-202` initial evidence data
3. `JEPA-203` validation
4. `JEPA-204` catalog expansion
5. `JEPA-205` paper-code linking
6. `JEPA-206` citation ranking upgrade
7. `JEPA-207` QA eval slice
8. `JEPA-208` metadata completeness checks
9. `JEPA-209` release workflow doc
10. `JEPA-210` benchmark comparison and tag prep

## Release Risks

- The biggest delivery risk is trying to automate evidence extraction before the schema is stable.
- The biggest quality risk is improving retrieval logic without adding a mode-specific regression check.
- The biggest scope risk is letting `v1.2.0` become a corpus-ingestion project instead of a skill-quality project.

## Recommended Cut Line

If scope needs to shrink, keep `JEPA-201` through `JEPA-209` and postpone any broader corpus automation. The minimum credible `v1.2.0` is:

- real evidence index
- paper-code links for core assets
- QA-specific eval improvement
- repeatable release checks
